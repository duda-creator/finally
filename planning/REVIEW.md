## 2. `.claude/agents/change-reviewer.md` (new)

This defines the very agent that produced this review, so treat the self-referential aspect as expected, not a defect.

```yaml
---
name: change-reviewer
description: This custom agent carries out a comprehensive review of all chanages since the last commit, providing feedback on code quality, structure, and potential issues or improvements.
---
```

### Issues

1. **Typo in `description`: "chanages" → "changes".** This field is what the orchestrator/dispatcher matches against when deciding whether to route a task to this agent, so a typo here is more than cosmetic — it's a small risk to discoverability/matching quality, and it's user-facing in agent-listing UIs. Easy fix.
2. **Description and body are near-duplicates.** The `description` frontmatter and the body prompt are almost word-for-word the same text. That's not wrong (frontmatter is for routing, body is the actual instructions), but there's an opportunity to make the `description` a tighter one-line summary (for matching/scanning) while keeping the fuller instructions in the body — right now a reader has to read the same paragraph twice.
3. **No scope/exclusion guidance.** The instructions say "review all changes since the last commit" but don't clarify:
   - Whether "since the last commit" means against `HEAD`, the merge-base with `main`, or the last pushed commit — in a repo with feature branches this is ambiguous (in this case it happened to be `main` itself, so it didn't matter, but it will on a branch).
   - Whether the agent should include untracked files (it should, and this run did, but that's inferred rather than stated).
   - Whether the agent should review its own defining file / the `.claude/` config directory, which is a slightly unusual recursive case worth calling out explicitly so future runs don't skip `.claude/` changes by assuming they're "not code."
4. **No `tools` or `model` frontmatter fields.** Other agent-style definitions in this ecosystem (and the SDK docs) support scoping tools/model per agent. For a review-only agent, restricting to read-oriented tools (Read/Grep/Glob/Bash for git commands, Write only for the output file) would reduce the blast radius if the prompt is ever manipulated by content in the diff being reviewed (e.g., a malicious `PLAN.md` edit instructing the reviewer to change permissions — this is exactly the kind of prompt-injection surface a review agent walks into by design, since it's told to read arbitrary changed files). Worth hardening given this agent's job is to read untrusted diff content.
5. **Hardcoded output path.** `Save the results to planning/REVIEW.md` always overwrites the same file with no versioning/append behavior. Fine for a single-user workflow, but if this agent runs repeatedly across multiple review cycles, prior review history is lost each time. Consider whether that's intended (a "current state" snapshot) or whether dated/append history would be more useful — for this project's stated multi-agent audit trail model, an append or dated-filename option seems more aligned with the "agents interact through files" philosophy stated in `CLAUDE.md`.

## 3. `.claude/commands/doc-review.md` (new)

```
Review the documentation file in the planning folder called $ARGUMENTS
and add questions, clarifications or feedback to a new section at the end,
along with any opportunities to simplify
```

### Issues

1. **No frontmatter at all.** Unlike the agent file, this slash command has no `description` or `argument-hint` metadata. Most slash-command conventions support a `description` field (shown in command listings/autocomplete) and an `argument-hint` (e.g. `<filename>`) so users know `/doc-review PLAN.md` is the expected invocation instead of guessing. Right now a user typing `/doc-review` with no args gets undefined behavior (the command body just says "called $ARGUMENTS" with nothing substituted).
2. **Mutates the target file in place with no safety net.** "add ... to a new section at the end" means this command edits planning docs directly. There's no instruction to preserve existing content, avoid duplicate "Feedback" sections on repeat runs (running `/doc-review PLAN.md` twice would presumably append two "Feedback" sections rather than update one), or to show a diff/confirmation before writing. Given PLAN.md is described in `CLAUDE.md` as "the key document" and "the shared contract" for multiple agents, a command that can silently accumulate duplicate sections in it over repeated runs is a minor but real maintainability risk.
3. **Ambiguous path resolution.** "the documentation file in the planning folder called $ARGUMENTS" assumes `$ARGUMENTS` is a bare filename (e.g., `PLAN.md`) rather than a path, and assumes the file lives directly in `planning/` rather than `planning/archive/` or elsewhere. Given the project already has `planning/MARKET_DATA_SUMMARY.md` and a `planning/archive/` folder (per `CLAUDE.md`), it's worth being explicit about how `$ARGUMENTS` maps to a path, and what happens if the file isn't found.
4. **No guidance on review depth/criteria.** "questions, clarifications or feedback ... along with any opportunities to simplify" is quite open-ended compared to the more structured `change-reviewer` agent (which asks for bullet points, refactoring callouts, best practices, etc.). For consistency between the two new review tools, consider aligning the level of structure/detail expected — right now `doc-review` reads like a quick first draft next to the more fleshed-out `change-reviewer.md`.
